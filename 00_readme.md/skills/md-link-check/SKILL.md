# Markdown Link & TOC Check Rules

## 목적

`.md` 파일 작성/수정 후 다음 세 가지를 반드시 검증합니다.

1. 내부 파일 링크 (`[텍스트](경로.md)`) — 파일 존재 여부
2. 앵커 링크 (`#섹션명`) — 실제 H2 섹션과 일치 여부
3. 목차 완결성 — 실제 H2 섹션이 목차에 모두 포함되어 있는지

---

## 1. 앵커 링크 검사

### GitHub 앵커 변환 규칙

헤딩 텍스트 → 앵커 변환 시 적용되는 규칙:

1. 소문자 변환
2. 영숫자·한글·공백·하이픈 이외 문자 **제거** (특수문자 포함)
3. 공백 → `-` 변환 (연속 공백도 각각 `-`로, 축약 없음)

특수문자 제거로 인한 이중 하이픈 발생 예시:

| 헤딩 | 앵커 | 이유 |
|------|------|------|
| `컨텍스트 / 도구 상세` | `#컨텍스트--도구-상세` | `/` 제거 → 양쪽 공백이 각각 `-` |
| `토큰 절약 & 성능 최적화` | `#토큰-절약--성능-최적화` | `&` 제거 → 이중 하이픈 |
| `` `@` 참조 기능 `` | `#--참조-기능` | 백틱·`@` 제거 → 앞 공백 `-` |
| `Plan — 에이전트` | `#plan--에이전트` | `—` 제거 → 이중 하이픈 |
| `CI/CD 비대화형` | `#cicd-비대화형` | `/` 제거, 양쪽 공백 없으면 단순 제거 |

### 앵커 검증 스크립트

```python
import re

def github_anchor(heading):
    s = re.sub(r'^#+\s*', '', heading.strip())
    s = s.lower()
    s = re.sub(r'[^\w\s\-\uAC00-\uD7A3]', '', s)
    s = re.sub(r' ', '-', s)
    return s
```

### 코드블록 내부 헤딩 주의

` ``` ` 안의 `## 헤딩`은 실제 섹션이 아닙니다.
목차 링크 대상은 **코드블록 밖의 H2만** 해당합니다.

## 2. 목차 완결성 검사

### 규칙

- 코드블록 밖의 모든 H2 섹션은 목차에 포함되어야 합니다.
- 단, 아래 고정 섹션은 목차에서 제외합니다: `목차`, `참고 자료`, `통계`, `참고 URL`
- 목차에 있는 링크가 실제 H2 섹션(코드블록 밖)을 가리키는지 확인합니다.

### 검사 절차 (문서 작성/수정 후 필수)

1. 코드블록 밖 H2 목록 추출
2. 각 H2의 올바른 앵커 계산 (`github_anchor()` 함수 사용)
3. 목차의 모든 링크 앵커와 대조 → 불일치 시 수정
4. 목차에 없는 H2 → 목차에 추가

## 3. 내부 파일 링크 검사

```bash
# 특정 파일의 내부 링크 검사
BASE=/path/to/repo
grep -oP '\[.*?\]\(\K[^)#]+\.md' FILE.md | while read link; do
  [ -f "$BASE/$link" ] && echo "OK  $link" || echo "❌  $link"
done
```

### 수정 규칙

1. 파일명 변경 반영 — 링크 텍스트와 경로 모두 수정
2. 경로 누락 — 서브디렉토리 경로 추가
3. 상대 경로 — 서브디렉토리에서 상위 참조 시 `../` 사용

---

## 4. 문서 작성 후 체크리스트

문서를 새로 작성하거나 H2 섹션을 추가/수정/삭제한 경우 반드시 확인합니다.

- [ ] 모든 H2 헤딩의 앵커를 `github_anchor()` 규칙으로 계산했는가
- [ ] 목차의 모든 링크가 코드블록 밖 실제 H2를 가리키는가
- [ ] 코드블록 안의 `## 헤딩`을 목차에 포함하지 않았는가
- [ ] 새로 추가한 H2 섹션을 목차에 추가했는가
- [ ] 특수문자(`/`, `&`, `—`, `@`, 백틱 등) 포함 헤딩의 앵커에 이중 하이픈(`--`)을 적용했는가
