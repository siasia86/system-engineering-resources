## 1. 개념 정리

| 용어            | 정의                                         | 예시                                       |
|-----------------|----------------------------------------------|--------------------------------------------|
| Edge case       | 유효 범위의 경계에서 발생하는 케이스         | 배열 첫/마지막, 0바이트 파일, 최대 연결 수 |
| Corner case     | 여러 경계 조건이 동시에 만나는 케이스        | 디스크 풀 + 네트워크 단절 동시 발생        |
| Boundary value  | 입력 범위의 min/max 값                       | 0, -1, MAX_INT, 빈 문자열                  |
| Degenerate case | 입력이 극단적으로 단순하거나 비어있는 케이스 | null, 빈 리스트, 요소 1개 배열             |
| Error case      | 의도적으로 잘못된 입력/상태                  | 잘못된 타입, 권한 없는 경로, 만료된 토큰   |
| Race condition  | 동시 실행으로 인한 타이밍 의존 충돌          | 파일 lock 없이 동시 쓰기, 중복 요청        |
| Happy path      | 모든 조건이 정상인 흐름 (비교 기준)          | 유효한 입력, 정상 네트워크, 충분한 리소스  |

### BVA와의 관계

경계값 분석(BVA)은 "수치 입력의 min/max ±1"에 집중합니다.
Edge case testing은 BVA를 포함하면서 환경, 상태, 시간, 리소스까지 확장합니다.

- BVA: [경계값 분석](../01_black_box/boundary_value_analysis.md) 참고.

## 2. 도출 방법

5가지 축으로 분류하여 체계적으로 도출합니다.

| 축     | 질문                           | 도출 예시                               |
|--------|--------------------------------|-----------------------------------------|
| 입력   | 빈 값, 최대 길이, 특수문자?    | null, 0, MAX_INT, 유니코드, 줄바꿈      |
| 환경   | 디스크, 메모리, 네트워크 제한? | 디스크 풀, OOM, DNS 실패, 저대역폭      |
| 상태   | 초기/중간/종료 상태 전이?      | 미초기화, 중간 실패, 재시작 직후        |
| 시간   | 동시성, 타임아웃, 순서 역전?   | 동시 쓰기, 만료된 세션, 역순 이벤트     |
| 리소스 | 고갈, 경합, 한계 도달?         | FD 고갈, connection pool 만원, PID 한계 |

### 도출 공식

```
테스트 케이스 = Happy path × (입력 경계 + 환경 제약 + 상태 전이 + 시간 경합 + 리소스 한계)
```

각 축에서 최소 1개씩 도출하면 기본적인 edge case 커버리지를 확보합니다.

## 3. 인프라 체크리스트

인프라 변경/운영 시 확인해야 하는 edge case입니다.

### 네트워크

| 상황             | 확인 항목                      |
|------------------|--------------------------------|
| 네트워크 단절    | 재연결 시 자동 복구되는가      |
| DNS 실패         | fallback IP, timeout 설정      |
| 패킷 손실 (1~5%) | 재시도 로직, 멱등성            |
| 대역폭 제한      | timeout 충분한가, 큰 파일 처리 |

### 디스크/스토리지

| 상황             | 확인 항목                       |
|------------------|---------------------------------|
| 디스크 풀 (100%) | 쓰기 실패 시 graceful 처리      |
| inode 고갈       | 소규모 파일 대량 생성 시나리오  |
| 읽기 전용 마운트 | 임시 파일 경로 대체 가능 여부   |
| NFS 지연         | lock timeout, stale file handle |

### 프로세스/서비스

| 상황                 | 확인 항목                      |
|----------------------|--------------------------------|
| 중복 실행            | file lock, PID 확인            |
| 좀비 프로세스        | 정리 로직, 재시작 시 잔존 처리 |
| OOM Kill             | 메모리 제한, cgroup 설정       |
| systemd restart loop | RestartSec, StartLimitBurst    |

### 권한/보안

| 상황                  | 확인 항목                       |
|-----------------------|---------------------------------|
| sudo 없이 실행        | 권한 부족 시 명확한 에러 메시지 |
| 토큰/키 만료          | 자동 갱신, 만료 전 경고         |
| SELinux/AppArmor 차단 | 감사 로그 확인, 정책 추가 방법  |

## 4. 스크립트 체크리스트

Python/Bash 스크립트 작성 시 확인해야 하는 edge case입니다.

### 입력 데이터

| 상황                   | 확인 항목                         |
|------------------------|-----------------------------------|
| 빈 문자열/None         | 조기 반환, 기본값 처리            |
| 한글/유니코드 파일명   | encoding 명시, display width 계산 |
| 특수문자 ($, backtick) | shell 이스케이프, 따옴표 처리     |
| 대용량 파일 (GB+)      | 메모리 매핑, 스트리밍 처리        |
| 줄바꿈 혼재 (LF/CRLF)  | universal newlines, strip 처리    |

### 파일 시스템

| 상황              | 확인 항목                      |
|-------------------|--------------------------------|
| 파일 없음         | FileNotFoundError 처리         |
| 빈 파일 (0 bytes) | 빈 결과 반환 vs 에러           |
| 심볼릭 링크       | 실제 경로 해석, 순환 링크 방지 |
| 경로에 공백 포함  | 따옴표 처리, pathlib 사용      |
| 동시 접근         | file lock, atomic write        |

### 동시성/병렬

| 상황                    | 확인 항목                       |
|-------------------------|---------------------------------|
| 동일 스크립트 중복 실행 | fcntl.flock / PID file          |
| 부모 프로세스 Kill      | 자식 프로세스 정리, signal 처리 |
| Ctrl+C 중단             | finally 블록, 임시 파일 정리    |

## 5. 테스트 방법

| 방법            | 용도                     | 도구/기법                         |
|-----------------|--------------------------|-----------------------------------|
| 수동 재현       | 일회성 검증, 복잡한 환경 | 의도적 디스크 풀, 네트워크 차단   |
| 자동화          | 반복 검증, CI 통합       | pytest parametrize, BVA 자동 생성 |
| Mock/Stub       | 외부 의존성 제거         | unittest.mock, monkeypatch        |
| Fault injection | 장애 상황 시뮬레이션     | kill -9, tc netem, stress-ng      |
| Fuzzing         | 예상 못한 입력 대량 생성 | AFL, hypothesis (Python)          |

### pytest 예시 (BVA + Edge case)

```python
import pytest

@pytest.mark.parametrize("value,expected", [
    (0, True),        # boundary: min
    (100, True),      # boundary: max
    (-1, False),      # edge: below min
    (101, False),     # edge: above max
    (None, False),    # degenerate: None
    ("", False),      # degenerate: empty string
    (50, True),       # happy path
])
def test_validate_score(value, expected):
    assert validate_score(value) == expected
```

### Bash 장애 주입 예시

```bash
# 디스크 풀 시뮬레이션
dd if=/dev/zero of=/tmp/fill_disk bs=1M count=10240

# 네트워크 지연 주입 (100ms + 25% 패킷 손실)
tc qdisc add dev eth0 root netem delay 100ms loss 25%

# CPU 부하 (4 코어 100%)
stress-ng --cpu 4 --timeout 60s

# 정리
rm /tmp/fill_disk
tc qdisc del dev eth0 root
```

## 6. 실전 사례

### md-style-check.py 중첩 박스 다이어그램 버그 (2026-06-19)

**증상:** 중첩 박스 다이어그램에서 내부 `└┘` 행의 폭 불일치를 감지하지 못했습니다.

**Edge case 유형:** corner case (중첩 구조 + 문자열 패턴 매칭 동시 발생)

**원인:**

```python
# Before: '┘' 가 행 내 아무 곳에 있으면 박스 종료로 판단
if '┘' in l:

# 내부 행 (외부 박스 종료 아님):
# │  └────┘  └────┘  └────┘   │  ← '┘' 포함하지만 종료 아님
```

**해결:**

```python
# After: 행이 └ 로 시작하고 ┘ 로 끝날 때만 외부 박스 종료
if l.strip().startswith('└') and l.strip().endswith('┘'):
```

**도출 축 분석:**

| 축   | 적용                                        |
|------|---------------------------------------------|
| 입력 | 중첩 박스 (box-in-box)                      |
| 상태 | 파서가 "박스 내부" 상태에서 내부 └┘ 를 만남 |
| 환경 | 해당 없음                                   |
