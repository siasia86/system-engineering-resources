## 2. 핵심 개념

### 컨텍스트 윈도우

AI가 한 번에 참고할 수 있는 정보량(토큰)에 제한이 있습니다.

```
┌──────────────────────────────────────────────────┐
│              Context Window (tokens)             │
│                                                  │
│  [system prompt] [memory] [history] [user input] │
│                                                  │
│  ████████████████████████████░░░░░░░░░░░░░░░░░░  │
│  ◄── used (70%) ──►       ◄── remaining ──►      │
│                                                  │
│  > 70% : precision degrades                      │
└──────────────────────────────────────────────────┘
```

### 메모리 (Memory)

세션 간 유지해야 하는 핵심 정보를 저장합니다. `/memory` 명령으로 관리합니다.

- 프로젝트 컨벤션
- 아키텍처 결정 사항
- 환경 정보 (IP, 계정, 경로)
- 작업 히스토리 요약

## 3. 디렉토리 구조

```
my-project/
├── .gstack/
│   ├── memory.md        ← 핵심 기억 (세션 간 유지)
│   ├── sessions/
│   │   ├── 2026-06-18_session1.log
│   │   └── 2026-06-18_session2.log
│   └── state.json       ← 세션 상태
├── src/
├── terraform/
└── ...
```

| 파일/디렉토리 | 역할                                      |
|---------------|-------------------------------------------|
| `.gstack/`    | gstack 작업 데이터 루트                   |
| `memory.md`   | `/memory` 명령으로 업데이트하는 영구 기억 |
| `sessions/`   | 세션별 대화 로그                          |
| `state.json`  | 현재 세션 상태 (휘발 가능)                |

## 4. 메모리 관리

### 언제 메모리를 업데이트 하는가

- 아키텍처 결정이 확정됐을 때
- 환경 정보가 변경됐을 때 (IP, 키, 경로)
- 컨벤션/규칙이 합의됐을 때
- 긴 세션 중 중간 정리가 필요할 때

### 메모리 업데이트 명령어

```
/memory
```

이 명령어를 실행하면 현재까지의 핵심 내용이 `memory.md`에 기록됩니다.

### 메모리 효율적 사용

```
# 좋은 메모리 내용 (구체적, 실행 가능)
- SSH 키: ~/.ssh/id_ed25519
- Docker 컨테이너 15대: 10.200.101.131~.157
- Terraform provider: AWS ~> 5.0, profile 01_re

# 나쁜 메모리 내용 (모호, 중복)
- 작업 진행 중
- 여러 가지 시도함
```

## 5. 주의사항

### 컨텍스트 제한 (70% 임계)

컨텍스트 윈도우의 70% 이상이 차면 답변 정밀도가 저하됩니다.

| 상태 | 사용률 | 대응                                     |
|------|--------|------------------------------------------|
| 정상 | ~50%   | 유지                                     |
| 주의 | 50~70% | 불필요한 대화 정리 고려                  |
| 위험 | 70%+   | 새 세션 시작, `/memory` 업데이트 후 전환 |

### 휘발성 변수

서버리스 환경 등에서는 프로세스 인스턴스마다 메모리 변수가 다르게 할당됩니다.

- 상태 저장이 필요한 경우 → 로컬 파일시스템에 영구 기록합니다.
- `state.json`은 세션 중 휘발될 수 있습니다.
- 핵심 정보는 반드시 `memory.md`에 기록합니다.

### 로컬 저장 주의

| 항목      | 저장 위치           | 지속성 |
|-----------|---------------------|--------|
| 세션 로그 | `.gstack/sessions/` | 영구   |
| 메모리    | `.gstack/memory.md` | 영구   |
| 상태 변수 | 프로세스 메모리     | 휘발   |

## 6. 활용 패턴

### 인프라 작업

```
1. 세션 시작 → memory.md 로드 (환경 정보 자동 참조)
2. 작업 진행 (Terraform 코드 작성, AWS CLI 실행)
3. 중간 결과 /memory 업데이트
4. 세션 종료 → 로그 자동 저장
```

### 장기 프로젝트

```
세션 1: 설계 → /memory (아키텍처 결정)
세션 2: 구현 → /memory (완료 항목, 이슈)
세션 3: 트러블슈팅 → /memory (해결 방법)
세션 N: 유지보수 → memory.md 참조하여 컨텍스트 복원
```

### `.gitignore` 권장 설정

```
# gstack 세션 로그 (선택적)
.gstack/sessions/

# 메모리는 커밋 가능 (팀 공유 시)
# !.gstack/memory.md
```
